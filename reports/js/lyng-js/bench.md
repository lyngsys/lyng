## Watch Items

- The executable array runtime baselines now include both dense-indexed and iterator-driven rows. On this run, `array-heavy.literal-indexed-runtime` measured `753.96` ns/work-unit and `array-heavy.iterator-runtime` measured `9754.59` ns/work-unit.
- The iterator-driven row warmed `23` feedback slots across `23` live sites with `3652` template bytes, so the benchmark captures real iterator lowering and VM state rather than only the older dense-element hot path.
- `string-heavy.concat-runtime` remains the lower-level string/runtime proxy at `183.83` ns/work-unit.
- `regexp-heavy.runtime` measured `47432.57` ns/work-unit while exercising global iteration, sticky state, match indices, and named-capture replacement through the shared matcher path.
- RegExp observability rows separate constructor compilation (`3815.86` ns/work-unit), stable default exec/test (`34958.65` ns/work-unit), named replacement (`12628.19` ns/work-unit), and legacy static accessor reads (`2137.47` ns/work-unit).
- `class-heavy.runtime` measured `2491.82` ns/work-unit while warming `31` feedback slots across `31` live sites with `9174` template bytes, covering private fields, static blocks, and `super` dispatch on the executable runtime path.
- `async-heavy.frontend` remains the only frontend-only async workload in the current benchmark surface. That is now a benchmark-shape gap rather than a known lowering/runtime hole, so follow-up work should add an executable async runtime row.
- `typed-array-heavy.runtime` measured `828.30` ns/work-unit while warming `33` feedback slots across `33` live sites with `3838` template bytes, covering ArrayBuffer-backed views and DataView byte traffic on the executable runtime path.
- The seeded accounting snapshot reports `1` promise job and `1` backing store so the retained runtime-accounting surface is exercised by real data. Retained RegExp payloads report as a distinct runtime domain, with payload bytes treated as a lower-bound estimate because the current regex backend does not expose all internally owned tables. Iterator-heavy evidence still lives in the executable array/iterator workload rows because iterator state is transient VM execution state rather than a retained post-run runtime record.

## Known Gaps

- This report complements the dedicated bytecode-density suite rather than replacing it. The density run remains the finer-grained instruction-shape view.
- The remaining frontend-only row does not report AST or sema heap residency. Its current memory row is limited to atom payload and the explicit absence of code-template or feedback state.
- Module-cache accounting remains a future retained-runtime domain. Post-run iterator-record rows are still zero because the current iterator state is transient to active VM execution, so the benchmarked array/iterator runtime rows are the authoritative memory/perf signal for that surface.
- RegExp payload accounting is currently a lower bound: it includes source text, retained UTF-16 source units, flag text, and the `regress::Regex` struct, but not the backend's private instruction vectors, class tables, and capture metadata allocations.
